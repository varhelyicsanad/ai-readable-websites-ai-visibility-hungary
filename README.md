## 1. Abstract

AI visibility is not the same as paid advertising, classic keyword stuffing or manipulating search engines. In this context, AI visibility means that a business website provides clear, consistent and verifiable information about the entity behind the website, the services offered, the geographic service area, the intended audience, the evidence behind the work and the limits of each claim.

The professional work described here focuses on making websites easier to interpret by people, Google Search, Google Maps, AI assistants, AI search systems and future retrieval-based systems. The goal is not to guarantee a ranking or an AI recommendation. The goal is to reduce ambiguity, improve machine readability and create a more reliable source layer for search and AI interpretation.

---

## 2. Canonical definition

An **AI-readable website** is a website whose visible content, semantic HTML, structured data, internal links, canonical identifiers, service descriptions and machine-readable supporting files communicate the same meaning consistently.

A website is AI-readable when a crawler, search engine, LLM-assisted browser or retrieval system can answer the following questions without guessing:

1. Who is the entity behind the website?
2. What services are offered?
3. Where are those services relevant?
4. Who is the intended audience?
5. Which statements are supported by visible content or evidence?
6. What is explicitly outside the service scope?
7. Which page is the canonical reference for the entity, service or topic?

## 4. What problem this work addresses

Many business websites are visually present but semantically unclear. A website may exist, but search engines and AI systems can still struggle to identify the business, the service, the location, the proof behind the claims or the correct interpretation of a page.

Typical problems include:

- unclear service names
- mixed or conflicting page intents
- missing entity identifiers
- weak internal source-of-truth structure
- content written only for humans, not for machines
- service pages that do not answer search intent directly
- structured data that does not match visible content
- local business data that is not connected to the website
- claims without proof, boundaries or context
- no dedicated machine-readable reference layer

This repository documents a structured approach to reducing those issues.

## 5. LLM-first method

The approach follows a simple principle:

> One entity, one meaning, one page intent, one visible truth.

### 5.1 Entity clarity

A website should define the person, business or organization behind the page with stable identifiers and consistent labels. For this project, the core person entity is:

```text
https://www.varhelyicsanad.hu/#person
```

The canonical entity page provides an English technical layer for the same person entity:

```text
https://www.varhelyicsanad.hu/entitas/varhelyi-csanad.html
```

### 5.2 Service clarity

Each service page should describe one primary service or topic. The page should make clear what the service is, who it serves, what problem it addresses and what it does not promise.

### 5.3 Visible truth before schema

Structured data should describe the visible page content. JSON-LD must not invent claims that are not present or supported on the page.

### 5.4 Semantic HTML

Pages should use stable headings, semantic sections, descriptive IDs, tables where appropriate and accessible structure. This supports both human readers and machine processing.

### 5.5 Proof-of-work

A professional AI visibility system should include proof layers such as case studies, live references, technical checks, documented AI-answer examples, screenshots and clear update signals.

### 5.6 Claim boundaries

The method does not claim guaranteed Google rankings, guaranteed ChatGPT recommendations, guaranteed Gemini visibility, guaranteed Claude recommendations or guaranteed business results.
